## How It Works In The Transpiler

```
Stage 1: Parse finds "edi" keyword
Stage 2: Splits into [base component] + [edit chain]
Stage 3: Resolver loads default component
Stage 4: Overrides layer on top of defaults
Stage 5: Outputs merged final component
```

```javascript
class WFLEditResolver {
  resolve(node) {
    // 1. Load base component defaults
    const base = registry[node.element];
    const defaults = base.styles.defaults;
    
    // 2. Check for edit chain
    if (node.edits.length > 0) {
      const overrides = {};
      
      node.edits.forEach(edit => {
        switch(edit.property) {
          case 'txt': overrides.text = edit.value; break;
          case 'col': overrides.color = edit.value; break;
          case 'bg':  overrides.background = edit.value; break;
          case 'rd':  overrides.borderRadius = edit.value; break;
          case 'shd': overrides.boxShadow = edit.value; break;
          case 'pd':  overrides.padding = edit.value; break;
          case 'ht':  overrides.height = edit.value; break;
          case 'fnt': overrides.fontFamily = edit.value; break;
          case 'wt':  overrides.fontWeight = edit.value; break;
          case 'sz':  overrides.fontSize = edit.value; break;
          case 'ani': overrides.animation = edit.value; break;
        }
      });
      
      // 3. Merge defaults + overrides
      return {
        ...defaults,
        ...overrides    // overrides win
      };
    }
    
    return defaults;
  }
}
```

## How The AI Uses This System

The AI basically thinks in **two steps:**

```
Step 1: Pick the closest default component
        "I need a navbar" → nav.gls

Step 2: List what's different from default
        "needs to be darker" → edi bg:#0a0a0a
        "different button text" → edi[btn] txt:"Get Access"
        "brand color" → edi[btn] bg:#6366f1
```

This means:
- ✅ AI writes **way less code**
- ✅ No hallucinating CSS properties
- ✅ Defaults handle the boring stuff
- ✅ Overrides handle the custom stuff
- ✅ Predictable, consistent output every time
